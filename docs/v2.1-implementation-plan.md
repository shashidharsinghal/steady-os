## Adjustments to PRD as Written

Three places where the implementation plan diverges from the PRD's framing:

### Split P0.1 into two distinct tracks

The PRD treats "channel commission parsing" as one item. It's actually two:

- **Track A** (1 day): Audit existing Swiggy parser, fix UI to render `—`
  not `₹0` for unsettled rows. Add `settlement_status` column.
- **Track B** (3 days): Build the Zomato parser. Currently a stub that
  throws `ParserNotImplementedError`. Requires a sample Zomato annexure
  file from the partner.

Track A is in Sprint 1. Track B should be scheduled when a Zomato sample
is available — likely late Sprint 2 or Sprint 3. Without a sample, Claude
Code would write a parser blind, which we agreed earlier is the wrong
move.

### Remove P3.2 and P3.3 from this release

Employees and Contractors shipped per partner confirmation. No work needed.
If the walkthrough surfaced specific gaps (CSV import, role templates),
those become an "Employees v2" decision and are not bundled here.

### Add deterministic insights (not in PRD)

The PRD reserves the Tier 2 slot but assumes nothing in it for v2.1. A
middle ground exists: ship 4-5 SQL-computed insights into that slot now.
"Saturday revenue ran X% above your 4-week Saturday average." "Tuesday is
your weakest day." These are computable, useful, free, and validate the
slot's UX before LLM costs enter the picture.

### Add P&L cross-check (not in PRD)

A small line on the dashboard showing "P&L cross-check: ₹X computed vs ₹Y
on franchise P&L · 99.2% match" catches parser bugs in either direction.
Cheap to add, high signal value.

## Risks and Mitigations

| Risk                                                                                 | Mitigation                                                                               |
| ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------- |
| Today's Rush bug is in the parser, not the dashboard                                 | Sprint 1 item #4 starts with diagnosis before code change                                |
| Daily nag becomes annoying and gets muted                                            | Idempotent (once/day max); muteable per partner; copy is helpful not nagging             |
| Mobile redesign uncovers responsive issues across other pages                        | Scope mobile work to dashboard only in v2.1; broader responsive audit is separate        |
| Deterministic insights feel generic                                                  | Each insight uses real data points specific to the period; no template strings           |
| Partner forgets to upload Swiggy file → channel economics still empty after Sprint 1 | Honest UI ("awaiting settlement file") plus daily nag for stale data — both ship in v2.1 |

## Execution Notes

- **Sprint 1 ships as a single PR** — small enough to review together;
  shipping piecemeal creates intermediate "still partially broken" states.
- **Sprint 2-4 ship per item** — each item is independent enough.
- **Don't pre-write Sprint 3-4 prompts yet.** Get Sprint 1 done, learn from
  it, then prompt-write for Sprint 2. Spec drift is real.
- **Document each sprint's actual outcome** in CLAUDE.md after merge so
  future Claude Code sessions know what changed.
